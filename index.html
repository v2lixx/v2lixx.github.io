FFmpeg

Cross-platform multimedia framework.

  • CVE pending Medium

    libavcodec/zmbv: heap buffer over-read in zmbv_decode_xor_32

    zmbv_decode_xor_32()'s size check covers only the motion-vector table, not the trailing XOR pixel data; with all-d=1 vectors and frames ≥ 800×600 (32bpp), per-block reads run past decomp_buf and the leaked heap bytes get XOR'd into the output — heap over-read with disclosure into rendered video.

    PR #22995 ↗

  • CVE pending Medium

    avformat/rtspdec: reject non-positive ANNOUNCE Content-Length

    In RTSP listen mode, an oversized Content-Length wrapped to -1 after being parsed as int; the path then fell through to av_malloc(0)'s 1-byte fallback and wrote a NUL at sdp[-1] — a 1-byte heap-buffer-underflow.

    PR #22902 ↗

  • CVE pending High

    libavcodec/rasc: heap use-after-free in the RASC decoder

    In decode_move(), a GetByteContext indexes into s->delta; a type-0 move replaces that buffer via av_fast_padded_malloc, so subsequent loop iterations read move-table fields out of the freed region — heap UAF read of attacker-controlled bytes.

    PR #22992 ↗

  • CVE pending High

    libavcodec/tdsc: heap buffer overflow in tdsc_load_cursor

    tdsc_load_cursor()'s CUR_FMT_MONO branch carried over a stride adjustment from the BGRA path, but the MONO inner loop already advances dst by a full cursor_stride; for non-32-aligned cursor_w the extra adjustment makes dst drift past the allocation each row — heap OOB write.

    PR #22994 ↗

libpng

Reference PNG library used by virtually everything that touches PNG/APNG.

  • CVE-2026-40930 Medium

    Chunk smuggling in push-mode APNG parser via unconsumed chunk body

    In push-mode APNG, png_push_read_chunk() cleared PNG_HAVE_CHUNK_HEADER on an ignored inter-frame chunk without consuming its body or CRC — so the next iteration re-read those unread bytes as a fresh [length][type], letting a benign carrier (e.g. tHRW) smuggle a fake fdAT into the frame-data path.

GStreamer

Pluggable multimedia framework shipped with most Linux desktops.

  • CVE pending High

    gstwavpackdec: integer overflow → ~4 GB heap buffer overflow

    32-bit size math 4 * block_samples * channels in gst_wavpack_dec_handle_frame() wraps to a tiny g_malloc, then libwavpack's subsequent memset uses the un-truncated 64-bit value — a ~4 GB linear heap write. Reachable on the second frame, since the decoder skips header re-validation after context init.

    work item #5069 ↗

libzip

Library for reading, creating and modifying ZIP archives.

  • Credited

    Integer underflow in PKWARE-DCL zip_source_stat

    pkware_decrypt() subtracted the 12-byte PKWARE header from comp_size during ZIP_SOURCE_STAT without a lower-bound check; a crafted entry with comp_size < 12 underflowed and returned a wrapped near-UINT64_MAX size as trusted metadata. Reported with reproducer test cases that prompted the upstream fix; acknowledged in libzip's THANKS file.

Chromium

Open-source web browser project (Chrome's upstream).

  • Credited

    Null pointer dereference in SkStrike::glyphIDsToPaths

    SkStrike::glyphIDsToPaths ignored prepareForPath's return value, so a compromised renderer sending readStrikeData with hasPath=false could leave a glyph poisoned (sentinel set, path null) and then crash the GPU process by referencing it from a DrawSlugOp.
